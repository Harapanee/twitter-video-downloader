ブラウザの進化とWebでできることの広がり

技術解説 Web技術

このツールを作っていて一番驚いたのが、「ブラウザだけでここまでできるのか」ということだった。動画のダウンロード、HLSストリーミングのパース、WebAssemblyによる動画変換、Service Workerでのキャッシュ制御。全部ブラウザの中で完結してる。5年前の自分がこれを聞いたら信じなかったと思う。今回は、ブラウザがどうやってここまで進化してきたのかを振り返ってみる。

静的HTMLの時代

90年代のWebは本当にシンプルだった。HTMLで文書を書いて、リンクで繋ぐ。それだけ。ページを移動するたびにサーバーから新しいHTMLが丸ごと送られてきて、ブラウザはそれを表示するだけの「文書ビューア」だった。

JavaScriptは1995年にNetscapeに搭載されたけど、初期のJSでできることは本当に限られてた。フォームの入力チェックとか、マウスカーソルに星をついてこさせるとか、そういう小ネタレベル。「アプリケーション」なんて概念はWebには存在しなかった。

Ajax革命

2005年頃に登場したAjax（Asynchronous JavaScript and XML）がWebの方向性を大きく変えた。ページ遷移なしでサーバーとデータをやりとりできるようになって、Google Mapsが出てきた時は衝撃だった。地図をドラッグするたびに新しいタイルが読み込まれて、ページのリロードが一切ない。「これ、Webアプリじゃん」と世界中の開発者が気づいた瞬間だったと思う。

ただ、Ajax時代のコードは正直カオスだった。jQueryのコールバック地獄、ブラウザ間の挙動の違い（IE対応...）、状態管理の概念がほぼなくてグローバル変数まみれ。動くけどメンテ不能みたいなコードが量産された時代でもある。

SPAの台頭

2010年代に入ると、SPA（Single Page Application）の考え方が主流になっていく。React、Angular、Vueといったフレームワークが登場して、ブラウザ側でUIのレンダリングを全部やる設計が一般的になった。

SPAのいいところは、ページ遷移が爆速になること。サーバーからJSON（データだけ）を受け取って、ブラウザ側でHTMLを組み立てる。ネイティブアプリに近い操作感をWebで実現できるようになった。

ただ、SPAにも問題は山ほどあった。初回読み込みが重い、SEOに弱い、JavaScriptが壊れたら何も表示されない。その辺の課題を解決するために、SSR（サーバーサイドレンダリング）とかSSG（静的サイト生成）とか、結局またサーバー側に戻っていくような流れも出てきた。Webの歴史は振り子みたいに行ったり来たりしてて面白い。

PWA -- Webアプリのネイティブ化

PWA（Progressive Web App）は、Webサイトをネイティブアプリっぽく使えるようにする仕組み。ホーム画面に追加できて、オフラインでも動いて、プッシュ通知も送れる。

PWAを支えてるのがService Workerで、これがまた強力。ブラウザとネットワークの間に入って、リクエストをインターセプト（横取り）できる。一度キャッシュしたリソースはオフラインでも返せるし、バックグラウンドで同期処理もできる。

このツールでもService Workerを使っていて、ffmpeg.wasmの約30MBのバイナリを一度キャッシュしたら次回以降はネットワーク不要で読み込めるようにしてる。初回ロードが重い問題の軽減策として、Service Workerなしではやってられなかった。

ブラウザに搭載された強力なAPI群

ここからが本題。最近のブラウザには、昔ならネイティブアプリでしかできなかったことを可能にするAPIが山ほど載ってる。自分がこのツールを作る中で実際に使ったものを中心に紹介する。

WebAssembly（Wasm）

C/C++/Rustで書いたコードをブラウザ上でほぼネイティブ速度で実行できる技術。別記事で詳しく書いたけど、うちのツールではffmpegをWasmにコンパイルしたffmpeg.wasmを使って、ブラウザ内で動画変換をやってる。これがなかったら、サーバーに動画を送らずに変換する仕組みは実現できなかった。

Web Workers

JavaScriptはシングルスレッドなので、重い処理を走らせるとUIがフリーズする。Web Workersを使えばメインスレッドとは別のスレッドで処理を動かせる。ffmpeg.wasmの変換処理をWeb Workerの中で実行することで、変換中もプログレスバーがちゃんと動くし、ユーザーが他の操作をしても画面が固まらない。地味だけど、これがないとUXが壊滅的になる。

Fetch API / Streams API

HLSの動画セグメントを並列ダウンロードするのにFetch APIは必須だった。XMLHttpRequestの時代と比べると、Promiseベースで書けるし、Streams APIと組み合わせればダウンロード進捗もリアルタイムで取れる。TSセグメントを数十本同時にfetchして、返ってきた順にバッファに貯めていく処理は、Fetch+Streamsの組み合わせがなければ相当面倒だったはず。

Blob / URL.createObjectURL

ブラウザ内で生成したバイナリデータ（変換後のMP4ファイル）をユーザーにダウンロードさせるために使う。BlobオブジェクトからURLを生成して、aタグのhrefに突っ込んでclick()する。サーバーを経由せずにファイルをダウンロードさせる基本テクニック。

IndexedDB

ブラウザ内のローカルデータベース。localStorageと違って、バイナリデータの保存もできるし容量も大きい。大容量の動画データの一時保存先として検討したことがある（結局メモリ上でやり切る設計にしたけど）。Webアプリでちゃんとしたデータ永続化が必要な場面では、IndexedDBは強い味方になる。

WebRTC

ブラウザ間のリアルタイム通信。ビデオ通話とかP2Pのファイル共有に使われてる技術。このツールでは直接は使ってないけど、Discordのブラウザ版やGoogle Meetが別途アプリのインストールなしで動いてるのはWebRTCのおかげ。ブラウザだけでビデオ会議ができるって、冷静に考えるとすごいことだ。

「ブラウザがOS」になる未来

最近のChromebookなんかは、まさにブラウザがOSそのものとして動いてる。そしてそれが成立してしまう程度には、ブラウザでできることが広がってる。

  • 文書作成・表計算: Google Docs / Sheets（もはやOffice並み）
  • 画像編集: Photoshop Web版（WebAssembly製）
  • コード開発: VS Code for the Web / GitHub Codespaces
  • 動画編集: Clipchampなどブラウザ完結の動画編集ツール
  • 3Dモデリング: WebGLベースのCADツール
  • ゲーム: Unity/Unreal EngineのWasm出力

File System Access APIを使えばローカルファイルの読み書きもできるし、Web Bluetooth、Web USB、Web Serialで外部デバイスとの通信すらブラウザから直接できる。正直、ネイティブアプリじゃないとできないことの方が少なくなってきてる。

自分がこのツールを最初に設計した時、「ブラウザ内でffmpegを動かして動画変換する」という構想を人に話したら、「それ無理でしょ」と言われた。でも実際にやってみると動いた。しかもそこそこ実用的な速度で。WebAssembly、Service Worker、Web Workers、Fetch API、Blob。これらのAPIを組み合わせたら、数年前なら「サーバーがないと無理」だったことがブラウザだけでできてしまった。

まだ残ってる課題

とはいえ、ブラウザの限界がゼロになったわけじゃない。

  • メモリ制限: ブラウザのタブに割り当てられるメモリには上限がある。大きなファイルを扱おうとすると落ちる。ネイティブアプリならOSのメモリをもっと自由に使える
  • GPUアクセス: WebGPUが策定中だけど、まだネイティブのGPU APIほどの自由度はない。ゲームや機械学習の推論ではここがボトルネックになる
  • ファイルシステム: File System Access APIは便利だけど、セキュリティ上の制約で使えない場面もある。特にSafariの対応が遅い
  • ブラウザ間の差異: API自体はあっても、ブラウザによって対応状況がバラバラなことがまだある。特にiOS Safari。開発者の永遠の悩み

まとめ

静的HTMLの文書ビューアだったブラウザが、Ajax、SPA、PWAと進化を重ねて、今やWebAssemblyで動画変換までこなすアプリケーション実行環境になった。自分でこのツールを作ってみて痛感したのは、ブラウザの進化って理論的に知ってるのと実際に使い倒すのでは全然印象が違うということ。「え、これブラウザだけで動くの？」という驚きが開発中に何度もあった。ブラウザでできることはまだまだ広がっていくと思うし、その進化の恩恵を活かしたツールをこれからも作っていくつもりです。
